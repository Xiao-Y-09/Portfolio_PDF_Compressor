## 第 1 章：系统定位与用户价值

### 1.1 系统是什么

面向设计学生和专业人士的 PDF 智能压缩工具：通过**页面分类 + 元素级压缩策略 + 二分搜索收敛**，将大体积作品集压缩到用户指定的目标大小，且保留矢量精度、文字清晰度、书签/超链接等结构化信息。

### 1.2 参考场景

设计学生作品集压缩：输入是含渲染图 / CAD / 文字混排的 PDF 作品集（典型 50-200MB），目标是压缩到 5 / 10 / 15 / 20MB 之内且保留可读性（申请学校/求职的常见硬性上限）。

### 1.3 与传统工具的区别

| 传统工具 | 本系统 |
|---------|--------|
| 暴露几十个参数（quality、DPI、色彩空间……） | 用户只选 3 样：目标大小 + 压缩场景 + 分类修改 |
| 全文档统一参数 | 页面分类 + 元素级差异化策略（面积感知） |
| 压出多大算多大 | 二分搜索收敛到目标大小 ±15%，失败明确报错 |
| 文字/矢量常被无差别栅格化 | 文字、矢量、书签、超链接结构化保留 |

### 1.4 核心设计原则（七条铁律）

1. **决策与执行分离**：Phase D（决策）不做压缩，Phase E（执行）不做决策
2. **Phase D 纯函数**：决策模块禁止文件 I/O、网络调用、图像处理库，只做数学运算
3. **只降不升**：不升采样、不升质量、不升 DPI
4. **面积感知**：压缩力度必须考虑 `area_ratio = element_bbox_area / page_area`
5. **临时文件强制清理**：上传的 PDF 处理完成后（无论成败）5 分钟内删除，永不持久化
6. **输入输出契约不可下游修改**：契约只能消费不能修改，调整必须回到契约层
7. **收敛失败不静默输出**：5 轮未达标必须明确报错，不允许输出超标 15% 以上的文件

## 第 2 章：三层物理架构

```
┌─────────────────────────────────────────────────────────┐
│  前端  Next.js 14 (Vercel)                               │
│  上传 / 目标选择 / 进度轮询 / 分类 review / 下载           │
└──────────────────────┬──────────────────────────────────┘
                       │ HTTPS (REST, JSON)
┌──────────────────────▼──────────────────────────────────┐
│  后端  Python 3.11+ / FastAPI (AWS EC2)                  │
│  ┌────────────┐  ┌──────────────────────────────────┐   │
│  │ API 层      │→│ Celery worker（压缩引擎主体）        │   │
│  │ upload      │  │ Phase A-F 全部流水线逻辑            │   │
│  │ compress    │  └───────────┬──────────────────────┘   │
│  │ tasks       │              │                          │
│  │ download    │       Redis（broker + result +          │
│  │ health      │              review session）           │
│  └────────────┘                                          │
└──────────────────────┬──────────────────────────────────┘
┌──────────────────────▼──────────────────────────────────┐
│  存储  EC2 本地 EBS（临时工作区，处理完立即清理）            │
│        + S3（可选，长期归档；s3_bucket 未配置则不启用）      │
└─────────────────────────────────────────────────────────┘
```

- **前端（Next.js on Vercel）**：用户上传、进度显示、分类 review、结果下载。技术栈：Next.js 14 App Router + TypeScript + Tailwind CSS + Zustand + Axios。
- **后端（Python/FastAPI on AWS EC2）**：压缩引擎主体，含 Phase A-F 全部逻辑。长任务经 Celery + Redis 异步化，API 层只做触发与查询。
- **存储（EC2 本地 EBS + S3 可选）**：临时文件保存在 `tmp_dir/<file_id>.<ext>`（file_id 为 UUID v4），retention 300 秒；S3 仅在 `aws.s3_bucket` 配置后启用，接口与本地存储一致（`get_storage()` 工厂选择）。

## 第 3 章：Phase 划分与数据流总览

### 3.1 构建 Phase（工程顺序）与运行 Phase（流水线）的对应

| 构建 Phase | 运行别名 | 模块 | 职责 |
|-----------|---------|------|------|
| Phase 4 | Phase A | `pipeline/split.py` | PDF 拆页 |
| Phase 5 | Phase B | `pipeline/extract.py` | 元素提取（位图/矢量/文字/字体/XObject/注释） |
| Phase 6 | Phase C | `pipeline/classify.py` | 页面分类（启发式 V1，可替换 AI V2） |
| Phase 7 | Phase D0 | `pipeline/preprocess.py` | 确定性预处理（字体子集化 + 固定开销计算） |
| Phase 8 | Phase D | `pipeline/decide.py` | 压缩参数决策（**纯函数**） |
| Phase 9 | Phase E | `pipeline/compress.py` | 压缩执行（唯一允许 PIL/OpenCV 之处） |
| Phase 10 | Phase F | `pipeline/assemble.py` | PDF 重组与输出 |
| Phase 11 | — | `pipeline/orchestrator.py` + `queue/tasks.py` | 流水线编排 + 收敛循环 + Celery 任务 |

### 3.2 流水线数据流图

```
input.pdf
   │
   ▼
Phase A 拆页 ──────────────► List[RawPage]
   │
   ▼
Phase B 元素提取 ──────────► List[PageElements] + Dict[str, FontUsage] + metadata
   │
   ▼
Phase C 分类 ──────────────► List[ClassifiedPage]
   │
   ⏸ 用户 review 断点（AWAITING_REVIEW，session 存 Redis，TTL 30 分钟）
   │   用户确认/修改分类、标记 skip/aggressive 页
   ▼
Phase D0 确定性预处理 ─────► PreprocessResult（子集化字体 + total_fixed_bytes）
   │                          （只跑一次）
   ▼
┌──────────── 收敛循环（最多 5 轮）─────────────┐
│  Phase D 决策（纯函数）──► CompressionPlan     │
│     │                                        │
│     ▼                                        │
│  Phase E 执行（真压缩）──► CompressionResult   │
│     │                                        │
│  total = raster_bytes + fixed_bytes          │
│  gap 在容忍区间内？ ──是──► 跳出               │
│     │否                                      │
│  previous_result 反馈给下一轮 Phase D          │
└──────────────────────────────────────────────┘
   │
   ▼
Phase F 重组 ──────────────► output.pdf（位图回填 bbox、矢量重绘、
   │                         文字+子集化字体、书签/超链接、清理后元数据）
   ▼
下载（download_id，下载后 5 分钟删除）
```

### 3.3 关键架构原则："确定性前置，不确定性留给循环"

字体子集化的结果是完全确定的（同样字体 + 同样字符集 → 输出永远一样），因此 Phase D0 前置只跑一次，使得：

1. Phase D 的位图预算是精确数字：`raster_budget = target_bytes - total_fixed_bytes`
2. 二分搜索的搜索空间从"整个 PDF"缩小到"位图部分"，收敛更快

### 3.4 任务进度分配（前端展示）

```
Phase A (split):        0% → 10%
Phase B (extract):     10% → 30%
Phase C (classify):    30% → 40%
【AWAITING_REVIEW】     40%（等待用户）
Phase D0 (preprocess): 40% → 50%
Phase D+E (converge):  50% → 90%（每轮 +8%）
Phase F (assemble):    90% → 100%
```

## 第 4 章：用户 review 断点的设计

### 4.1 为什么需要断点

- 分类器（Phase C）不需要百分百准确，用户在 review 界面手动修正
- **光栅化是不可逆操作**（矢量→位图），必须让用户明确知情并有机会阻止
- 用户可对特定页面标记 `skip`（完全不压）/ `aggressive`（激进压）/ `reclassify`（改类型）

### 4.2 实现机制

Orchestrator 拆成两段：

```
run_until_classify(file_id, user_prefs, ctx, on_progress)
    → (classified, session_id)
    中间状态（page_elements、fonts、metadata、classified）序列化
    保存到 tmp_workspace/session.pickle；session 元数据存 Redis：
    key = "session:{task_id}"，TTL = 1800 秒

resume_after_review(session_data, user_modified_classified, user_prefs, on_progress)
    → 最终 PDF 路径
    从 pickle 恢复状态，继续 Phase D0 → F
```

### 4.3 任务状态机

```
PENDING → PROGRESS → AWAITING_REVIEW → PROGRESS → SUCCESS
                │                          │
                └────────► FAILURE ◄───────┘
```

- 前端每 2 秒轮询 `GET /api/v1/tasks/:task_id`
- `AWAITING_REVIEW` 返回 `{ classified, session_id }`，前端渲染分类网格；`complexity_score` 高的页面高亮"请确认"
- 用户确认后 `POST /api/v1/tasks/:task_id/resume` 触发后半段任务

### 4.4 超时策略

30 分钟内用户未响应 → 定时任务扫描过期 session → 清理对应 `tmp_workspace` + 删除 Redis session。用户回来看到任务已过期，需重新上传。

## 第 5 章：错误处理与临时文件清理策略

### 5.1 统一错误类型

所有 Phase 的错误走契约类型 `PhaseError { phase, code, message, recoverable }`（定义见《数据契约.md》）。已定义错误码：

| code | phase | 含义 | recoverable |
|------|-------|------|-------------|
| `INVALID_PDF` | split | 输入不是有效 PDF | false |
| `ENCRYPTED_PDF` | split | 加密 PDF | false |
| `TARGET_TOO_SMALL` | decide | 固定开销已超过目标大小 | false |
| `CONVERGENCE_FAILED` | orchestrator | 5 轮未收敛且超标 >15% | false |
| `ASSEMBLY_FAILED` | assemble | 输出 PDF 保存失败 | false |

### 5.2 错误容忍分级

- **单元素/单页失败 → 容忍**：单页提取失败记 warning 返回空 PageElements；单张图压缩失败用原图凑数（`compressed_bytes = original_bytes`）；字体无法加载则跳过子集化保留原字体
- **整体质量不达标 → 报错**：压缩整体成功率 < 80% 抛 PhaseError
- **收敛失败 → 明确报错**（铁律 7）：返回"目标 X MB 过小，当前最优压缩结果为 Y MB。建议选择更大的目标或标记更多页面为可激进压缩"，绝不静默输出超标文件
- **API 层**：PhaseError → HTTP 4xx/5xx + JSON body；未捕获异常 → 500，log 完整堆栈但不返回给用户

### 5.3 临时文件生命周期（铁律 5）

| 文件 | 位置 | 清理时机 |
|------|------|---------|
| 用户上传的 PDF | `tmp_dir/<file_id>.pdf` | retention_seconds = 300 秒后由定时清理任务删除 |
| 工作区（拆页/图片/字体/session.pickle） | `tmp_workspace/` | 任务成功：产出下载文件后立即 `cleanup_workspace()`；任务失败：except 分支同样清理；review 超时：定时扫描清理 |
| 压缩产物（供下载） | storage | 下载完成后 5 分钟删除 |

三重保障：
1. 任务级 try/finally 清理（成功失败都走）
2. FastAPI startup 启动的定时任务，每 60 秒调用 `storage.cleanup_expired()`
3. review session TTL（1800 秒）+ 过期扫描

**验收标准**：触发 10 个任务（含 3 个模拟失败），5 分钟后 `tmp_workspace` 全部为空。

## 第 6 章：AWS EC2 部署拓扑

### 6.1 为什么是 EC2

CPU 密集型的 PDF 处理任务需要完整 IaaS 控制权：精细调优 CPU/RAM、EBS 大盘存临时文件、细粒度安全组。性价比优于 Railway / Vercel Serverless。

### 6.2 拓扑图

```
用户 ──HTTPS──► Vercel（前端静态 + SSR）
   │
   └─HTTPS──► EC2 弹性 IP
                │
        ┌───────▼────────┐
        │ Nginx (443/80) │  SSL 终结（Let's Encrypt / certbot）
        │  client_max_body_size 500M
        │  proxy_read_timeout 300s
        └───────┬────────┘
                │ upstream api:8000
        ┌───────▼────────────────────────────┐
        │ docker-compose stack               │
        │  api            (uvicorn :8000)    │
        │  celery_worker  × 2 副本            │
        │  redis          (redis:7-alpine,   │
        │                  named volume 持久化)│
        └───────┬────────────────────────────┘
                │
        EBS gp3 100GB（tmp 工作区） + S3（可选归档，走 IAM Role）
                │
        CloudWatch Agent（/var/log/pdfcompress/*.log + metrics）
```

### 6.3 实例与资源配置

| 项 | 配置 |
|----|------|
| 实例类型 | 起步 t3.large（2 vCPU / 8GB）；重度 c6i.xlarge（4 vCPU / 8GB，CPU 优化） |
| EBS | gp3 100GB（临时工作区） |
| 安全组 | 22（仅限自己 IP）、80/443（公网） |
| IAM Role | 启用 S3 时授予 S3 读写权限（密钥不落盘） |
| 进程管理 | docker-compose（api / celery_worker / redis / nginx）；宿主机 systemd 管 docker 服务自启 |

### 6.4 监控与日志

- CloudWatch metrics：CPU、内存、磁盘、网络
- CloudWatch alarms：磁盘 > 80%、CPU > 90% 持续 5 分钟
- 应用日志：logrotate 每天，保留 7 天；CloudWatch Agent 收集 `/var/log/pdfcompress/*.log`
- 自定义业务指标：任务成功率、平均收敛轮数、平均处理时长

### 6.5 密钥与安全

- 密钥只走环境变量 / IAM Role，永不进代码、镜像、config.yaml
- `deploy/.env`、`deploy/ec2/certs/` 在 .gitignore
- API 层校验：Content-Type 为 application/pdf + 首字节 `%PDF` + 大小 ≤ 500MB（超限返回 413）

### 6.6 CI/CD（建议，不在核心范围内实现）

GitHub Actions：push to main → 构建镜像 → 推送 ECR → SSH 到 EC2 → `docker-compose pull && restart`。
