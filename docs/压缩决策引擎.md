# 压缩决策引擎（Phase D 规格）

> **地位**：本文档是项目三份真源文档之一，是 `backend/app/pipeline/decide.py`（构建 Phase 8）的唯一实现依据。
>
> **母本**：《PhaseD_压缩参数决策引擎.md》（详细规格）+《PDF压缩SaaS构建手册.md》第二节（三主控变量框架）、Phase 8 Prompt、附录 B。本文档是母本的结构化正式版，未新增任何母本之外的功能。
>
> **定位**：Phase D 是整个压缩流水线唯一包含"不确定性"的环节。其他 Phase 都是单程执行，Phase D 需要通过反馈循环收敛到目标文件大小。它是**纯函数**——相同输入永远产出相同 Plan，不做任何实际压缩，不读写磁盘。

## Phase D 的五条局部铁律

1. **纯函数**：不做任何实际压缩、不读写文件、不调用网络、不调用图像处理库（PIL/OpenCV/fitz），只做数学运算和 config 读取
2. **只降不升**：任何参数调整不能让元素变得比原始更大（不升采样、不升质量、不升 DPI）
3. **面积感知**：压缩力度必须考虑元素在页面中的视觉占比，小元素可以被更激进地压缩
4. **用户意志优先**：用户标记"不压缩"（skip）的页面，Phase D 不生成任何 plan，循环中也不触碰
5. **预算守恒**：所有页面的位图估算体积之和 ≤ raster_budget × 1.1，不允许超支

## 第 1 章：设计哲学 — 三主控变量框架

### 1.1 本质：把二三十个参数收敛为三个旋钮

传统 PDF 压缩工具暴露几十个参数（quality、DPI、色彩空间、下采样算法……），普通用户不知道怎么调。本系统的核心设计：**所有具体参数都从三个语义化的主控变量推导**。

### 1.2 三个主控变量

**变量 1：`aggressiveness`（压缩激进度，0.0 ~ 1.0）**

```
0.0 = 最保守，几乎不压
0.5 = 中等（默认，屏幕查看质量）
1.0 = 最激进，能压多狠压多狠
```

所有 quality、DPI 基准值从这一个数推导：

```
base_quality = lerp(quality_ceiling, quality_floor, aggressiveness)
base_dpi     = lerp(dpi_ceiling, dpi_floor, aggressiveness)
```

**变量 2：`area_sensitivity`（面积敏感度，0.0 ~ 1.0）**

控制"小图比大图压得狠多少"：

```
effective_aggressiveness = aggressiveness + area_sensitivity × (1.0 - area_ratio)

# area_ratio = 1.0 的满页主图：不加压
# area_ratio = 0.1 的小图：额外加 0.9 × area_sensitivity 的激进度
```

**变量 3：`pixel_ceiling_ratio`（像素天花板比率）**

根据元素在页面上的**显示尺寸**反算所需像素上限，超出部分强制裁掉。这是压缩效果最立竿见影的手段——一张 4K 渲染图缩小显示在页面角落时，多余像素直接砍掉。

```
display_size_inches = bbox / 72          # PDF 坐标单位是 point（1/72 英寸）
max_needed_pixels   = display_size_inches × target_dpi × pixel_ceiling_ratio
```

`pixel_ceiling_ratio = 1.2` 意味着允许比理论需要多 20% 的像素作为安全余量。

### 1.3 用户体验层的映射

用户在前端只选三样：**目标大小 + 压缩场景 + 分类修改**，后端映射为：

```
target_size_mb:      用户直选 5 / 10 / 15 / 20
compression_target:
  print  → aggressiveness = 0.2
  screen → aggressiveness = 0.5（默认）
  email  → aggressiveness = 0.8
per_page_override:   用户可对特定页面标记 skip / aggressive / reclassify
```

### 1.4 Phase D 在流水线中的位置

```
Phase A  拆页
Phase B  元素提取
Phase C  分类
         ⏸ 用户 review
Phase D0 确定性预处理（字体子集化 + 固定开销计算）  ← 只跑一次
Phase D  压缩参数决策（纯函数，可被反复调用）        ← 本文档
Phase E  压缩执行
         D+E 收敛循环（binary search）
Phase F  PDF 重组 + 输出
```

## 第 2 章：输入契约（DecisionInput）

Phase D 主入口签名（各类型的 Pydantic 定义见《数据契约.md》）：

```python
decide_compression(
    classified: List[ClassifiedPage],          # 来自 Phase C + 用户 review
    pages: List[PageElements],                 # 来自 Phase B（读取 dpi/bbox 等）
    user_prefs: UserPreferences,               # 用户选择
    preprocess: PreprocessResult,              # 来自 Phase D0（固定开销）
    previous_result: Optional[CompressionResult],  # 上一轮结果（第 1 轮为 None）
    round_number: int,                         # 当前轮数，从 1 开始
    config: CompressionConfig,                 # settings.compression
) -> CompressionPlan
```

输入语义（对应母本 DecisionInput）：

| 输入 | 来源 | 用途 |
|------|------|------|
| `classified` | Phase C + 用户 review | 页面类型影响基准参数；complexity_score 仅供前端 |
| `user_prefs.target_size_mb` | 用户 | 目标大小 5/10/15/20 MB |
| `user_prefs.compression_target` | 用户 | screen / print / email → 基础激进度 |
| `user_prefs.per_page_overrides` | 用户 review | skip / aggressive / reclassify |
| `preprocess.total_fixed_bytes` | Phase D0 | 矢量 + 文字 + 子集化字体 + 元数据 + PDF 结构开销 |
| `preprocess.per_page_fixed_bytes` | Phase D0 | 每页固定开销明细 |
| `pages[*].raster_images[*]` | Phase B | width/height/dpi/has_alpha/bbox |
| `pages[*].vector_paths[*]` | Phase B | control_point_count/bbox |
| `previous_result` | 上一轮 Phase E | total_raster_bytes + 每图实际结果；第 1 轮为 None |
| `round_number` | Orchestrator | 1..max_convergence_rounds |

**约束**：`previous_result = None` 时（首轮）必须使用基准推导，不崩溃。

- 所有 `quality ∈ [quality_floor, quality_ceiling]`（默认 [30, 95]）
- `has_alpha=true` 的图 `output_format = "png"`
- `Σ estimated_bytes ≤ raster_budget × 1.1`
- 所有字段均为《数据契约.md》已定义字段，不得私自扩展

## 第 4 章：核心公式

### 4.1 工具函数

```
lerp(a, b, t)   = a + (b - a) × t          # t ∈ [0,1] 线性插值
clamp(x, lo, hi) = max(lo, min(hi, x))
```

### 4.2 面积占比系数（area_ratio）

```
area_ratio = element_bbox_area / page_area
element_bbox_area = bbox.w × bbox.h
page_area         = page_width × page_height
```

**为什么需要它**：同一张 3000×2000 渲染图——铺满 A3 页面（area_ratio ≈ 0.95）用户会仔细看，需要高质量；作为角落缩略图（area_ratio ≈ 0.05）用户不会放大看，可以狠压。

| area_ratio 范围 | 含义 | 质量倾向 |
|----------------|------|---------|
| > 0.6（大图阈值） | 主图，占据页面大部分 | 保守，优先保质量 |
| 0.2 ~ 0.6 | 中等元素 | 中等压缩 |
| < 0.2（小图阈值） | 小图/缩略图/装饰 | 激进，大幅压缩 |

### 4.3 有效激进度推导（规范实现形式）

```python
eff_aggr = clamp(base_aggr + config.area_sensitivity * (1.0 - area_ratio), 0.0, 1.0)

quality = int(lerp(config.quality_ceiling, config.quality_floor, eff_aggr))
dpi     = lerp(config.dpi_ceiling, config.dpi_floor, eff_aggr)
```

> **与母本离散修正表的关系**：母本《PhaseD_压缩参数决策引擎.md》§5.3 给出的分档修正（小图 dpi_factor=0.5 / 中图 0.75 / 主图 1.0；quality_penalty = (1-area_ratio)×15）是本连续公式的行为参考——连续公式在默认常量（area_sensitivity=0.6）下应产生与分档表同方向、同量级的差异化。**实现采用连续公式**（单一机制、无分档跳变、完全由 config 常量驱动）；分档表用于人工校验决策输出是否合理。

> **决策理由记录（2026-07-04，用户批准）**：最终采用连续公式为规范实现、离散表降级为校验参考。理由：**避免临界跳变**——离散分档会让 area_ratio 恰好跨过 0.2 / 0.6 边界的两张相近图片得到突变的参数；连续公式由 config 常量驱动，行为**更可预测、可调**。

### 4.4 只降不升（铁律 3）

```python
target_dpi = min(dpi, image.dpi)          # 不升 DPI
# 收敛调整时同样约束：new_dpi = min(prev_dpi * dpi_factor, original_dpi)
# quality 永不超过 quality_ceiling，也永不高于让文件变大的程度
```

### 4.5 像素天花板

```python
display_w_inches = image.bbox.w / 72
display_h_inches = image.bbox.h / 72
max_w = display_w_inches * target_dpi * config.pixel_ceiling_ratio
max_h = display_h_inches * target_dpi * config.pixel_ceiling_ratio

if image.width > max_w:
    # 原图像素已超天花板，进一步降 DPI
    target_dpi = min(target_dpi, max_w / display_w_inches)
```

`max_dimension`（RasterPlan 字段）= 按天花板算出的最长边像素上限，作为 Phase E 的兜底保护。

## 第 5 章：渲染图（Raster）压缩策略

### 5.1 参数维度

按影响力排序：

1. **DPI 降采样**（影响最大）：300dpi → 150dpi，像素数变为 1/4，体积约变为 1/4
2. **JPEG 质量**（影响中等）：q=95 → q=70，体积约变为 1/3
3. **最大尺寸 max_dimension**（兜底保护）：防止超大图即使降了 DPI 依然过大

### 5.2 基准参数表（校验参考）

母本给出的 `compression_target × page_type` 基准参数（area_ratio = 1.0 满页主图时的期望值）：

```
                    screen          print           email
                    dpi   q         dpi   q         dpi   q
photo_heavy         150   80        250   90        96    65
mixed               150   78        250   85        96    62
text_heavy(小配图)   120   75        200   85        96    60
```

实现上这些值由第 4 章公式从 `presets[target].aggressiveness` + config 边界推导（例如 screen：aggressiveness=0.5 → base_quality = lerp(95, 30, 0.5) ≈ 62 起步，主图 area_ratio≈1 时 eff_aggr≈base，接近表中量级）。本表作为决策输出合理性的人工校验基准：若推导结果与表中数值方向或量级明显背离，说明常量配置或公式实现有误。

### 5.3 面积修正

采用第 4.3 节连续公式。期望行为（母本分档表）：

```
area_ratio < 0.2  → DPI 约减半、quality 额外下调（最多 -15）
area_ratio 0.2~0.6 → DPI 约 ×0.75、quality 小幅下调
area_ratio > 0.6  → 基准参数，主图保质量
quality 永不低于 quality_floor（30）
```

### 5.4 透明通道处理（修订 2026-07-04：三态 alpha_type）

```python
if image.alpha_type in ("none", "opaque"):
    output_format = "jpeg"    # 无 alpha / 伪透明（SMask ≥99% 为 255）→ JPEG，压缩率远高于 PNG
else:  # "translucent"
    output_format = "png"     # 真透明必须 PNG（JPEG 不支持透明）
    # PNG 压缩空间有限，主要靠降 DPI / 降分辨率
    # quality 对 PNG 无意义，执行层使用 png compress_level
```

**修订原因（生产数据驱动）**：三份真实作品集样本实测 ~97% 位图带 SMask。若按旧规则
"有 alpha 一律 PNG"，PNG 估算系数（0.35 B/px）≈ JPEG 的 3 倍，email/5MB 类激进目标
不可收敛。Phase B 在 Producer 层用像素采样区分伪透明（opaque）与真透明（translucent），
Phase D 只对真透明付 PNG 代价。`has_alpha ≡ (alpha_type == "translucent")` 保持兼容，
本文档所有 "has_alpha → png" 的表述仍然成立。

> **决策记录：flatten 策略暂缓（2026-07-04，用户裁决）**
> 实测发现按唯一 SMask 计仍有 52.6% 是重度真透明（过半像素 <255），曾评估两个方案：
> ① "translucent → 压平到白底/页面背景 → 转 JPEG"；② "合并同位置图层再 flatten"。
> **当时为什么没做**：
> - 主要体积杠杆不在此：97% 的 translucent 摆放来自少数被平铺数千次的贴图，
>   "每 xref 一份压缩流 + 多处共享插入"（见《数据契约.md》§3.1 共享 data_ref 设计）
>   已把 PNG 代价从"每摆放一次"降到"每唯一图一次"，收益先兑现且零风险；
> - flatten 是不可逆的视觉变更（透明叠加处背景色可能错），需要用户可见的开关与
>   review 提示，属产品决策不宜引擎自作主张；
> - 方案②（合并同位置图）收益不确定、实现风险高，且被共享插入方案替代。
> **重启条件**：Phase 8 在真实作品集上跑收敛，若 email/5MB 档因 PNG 体积无法收敛，
> 再评估引入"用户可选 flatten 开关"（方案①），方案②维持不做。

### 5.5 体积估算公式

Phase D 不做真正压缩，但需要估算体积做预算分配（粗略估算，误差 ±20% 可接受，binary search 会修正）：

```python
def estimate_raster_size(width, height, original_dpi, target_dpi, quality, fmt):
    scale = (target_dpi / original_dpi) ** 2
    pixel_count = width * height * scale

    if fmt == "jpeg":
        # JPEG 经验公式：每像素字节数随 quality 近似线性
        bytes_per_pixel = 0.04 + (quality / 100) * 0.12   # q=100→~0.16, q=50→~0.10
    else:  # png
        bytes_per_pixel = 0.35                             # PNG 无损，经验均值

    return int(pixel_count * bytes_per_pixel)
```

## 第 6 章：矢量图（Vector）压缩策略

### 6.1 矢量的特殊性与三种策略

矢量数据大多数情况已经很小，**默认策略是 keep（不动）**。例外是复杂 CAD/线稿图（数十万控制点），矢量数据本身可能比等效位图还大。

**策略 A：keep（保持原样）**

```
适用：面积小的矢量装饰（图标、箭头、边框）、控制点少的简单图形
触发：control_points < vector_simplify_threshold(10000) 或 area_ratio < 0.3
体积变化：0
```

**策略 B：simplify（路径简化）**

```
适用：中等复杂度矢量图，保留矢量格式但减少控制点
触发：10000 ≤ control_points < vector_rasterize_threshold(100000) 且 area_ratio ≥ 0.3
参数 tolerance：
  area_ratio > 0.6（主图）→ 0.5   # 几乎看不出变化
  area_ratio 0.3~0.6     → 1.0   # 轻微简化
风险：直线可能轻微弯曲，曲线可能失真
体积减少：约 20%-50%（估算按原始的 50% 保守取值）
```

**策略 C：rasterize（光栅化）**

```
适用：极端复杂矢量（超密 CAD 平面图），矢量数据比等效位图还大
触发：control_points ≥ vector_rasterize_threshold(100000) 且 area_ratio ≥ 0.3
参数 rasterize_dpi：
  area_ratio > 0.6   → 200
  area_ratio 0.3~0.6 → 150
  area_ratio < 0.3   → 96（该档实际不会触发，见 6.3 面积门槛）

⚠ 光栅化是不可逆的！用户必须在 review 界面被明确告知。
```

### 6.2 复杂度判断

复杂矢量的判定信号（母本 §6.1）：

```python
total_control_points > 50000
或 (total_control_points > 10000 且 vector_area_ratio > 0.7)
```

决策实现以控制点三级阈值为准（keep / simplify / rasterize 的切换点分别是
`vector_simplify_threshold` 和 `vector_rasterize_threshold`），上式用于 Phase C
分类与 review 界面提示"本页含复杂矢量"。

### 6.3 面积门槛（决策顺序：先面积，后复杂度）

```
area_ratio < vector_ignore_area_ratio(0.1) → 无条件 keep（太小不值得处理）
area_ratio < 0.3                           → 只能 keep 或 simplify（小图不值得 rasterize）
area_ratio ≥ 0.3                           → 按控制点三级阈值三选一
```

### 6.4 矢量体积估算

```python
def estimate_vector_size(vector, strategy, rasterize_dpi=None):
    if strategy == "keep":
        return vector.original_bytes
    elif strategy == "simplify":
        return vector.original_bytes * 0.5        # 保守估计减半
    elif strategy == "rasterize":
        # 光栅化后走位图估算（quality=80, jpeg）
        return estimate_raster_size(
            bbox.w * rasterize_dpi / 72,          # 宽（像素）
            bbox.h * rasterize_dpi / 72,          # 高（像素）
            rasterize_dpi, rasterize_dpi,
            quality=80, fmt="jpeg")
```

## 第 7 章：预算分配算法

### 7.1 预算计算（修订 2026-07-04：基于不可削减下限）

> **决策记录（用户批准）**：Phase 8 的 raster_budget 有效性检查应基于**不可削减部分**
> （文字 + 字体 + 元数据 + PDF 结构 + 矢量光栅化后下限成本），而非全部 fixed 开销。
> 否则复杂矢量作品集会在有 rasterize 策略可用的情况下被误判为 TARGET_TOO_SMALL。
> 实现细化：矢量策略由阈值规则决定、与预算无关，因此"矢量计划成本"即当前规则下的
> 可达下限（keep-only 组的下限就是 keep 成本）——下限在 decide 内部计算，无需新契约字段。

```python
target_bytes      = user_prefs.target_size_mb * 1024 * 1024
irreducible_fixed = preprocess.total_fixed_bytes - Σ per_page.vector_bytes
                    # = 文字 + 注释 + 子集化字体 + 元数据 + PDF 结构（真正不可削减）
vector_planned    = Σ 每个矢量组按已选策略的估算成本（keep/simplify/rasterize，见第 6 章）

raster_budget = target_bytes - irreducible_fixed - vector_planned

if raster_budget <= 0:
    raise PhaseError(phase="decide", code="TARGET_TOO_SMALL",
                     message=f"不可削减开销 {irreducible_fixed + vector_planned} 已超过目标 {target_bytes}",
                     recoverable=False)
```

### 7.2 按页分配（按视觉重要性加权，非平均）

```python
def compute_page_weight(page):
    """页面权重 = 该页位图原始体积 × 面积占比的加权和"""
    return sum(img.original_bytes * img.area_ratio for img in page.raster_images)

total_weight = sum(compute_page_weight(p) for p in pages if not p.skip)

for page in pages:
    if page.skip:
        page.budget = page.original_raster_bytes   # 不压缩，原样计入
    else:
        page.budget = raster_budget * compute_page_weight(page) / total_weight
```

### 7.3 页内分配

一页多张位图时，按 area_ratio 加权分配该页预算：

```python
for img in page.raster_images:
    img.budget = page.budget * (img.area_ratio / sum_area_ratios_in_page)
```

### 7.4 预算校验与再平衡（生成 Plan 的最后一步）

```python
total_estimated = sum(所有 RasterPlan.estimated_bytes)

if total_estimated > raster_budget * 1.1:
    # 估算已超预算 → 全局线性缩放 quality，直到估算总和 ≤ raster_budget
    scale = raster_budget / total_estimated
    for plan in all_raster_plans:
        plan.quality = max(config.quality_floor, int(plan.quality * scale))
        plan.estimated_bytes = 重新估算(plan)
```

## 第 8 章：Binary Search 收敛循环

### 8.1 循环结构（Orchestrator 侧，本章为 Phase D 提供上下文）

```
previous_result = None
for round_num in 1..config.max_convergence_rounds:      # 默认 5
    plan   = decide_compression(..., previous_result, round_num, config)  # Phase D
    result = execute_compression(plan, ...)                               # Phase E

    total     = result.total_raster_bytes + preprocess.total_fixed_bytes
    target    = user_prefs.target_size_mb * 1024 * 1024
    gap_ratio = (total - target) / target

    if -convergence_tolerance ≤ gap_ratio ≤ 0:        break   # 达标（略保守）
    if 0 < gap_ratio ≤ convergence_tolerance / 2:     break   # 微超标，接受

    previous_result = result
```

### 8.2 Phase D 在第 N 轮（N > 1）的调整逻辑

主控调整量是 **base_aggressiveness**（所有参数从它推导，调它 = quality 和 DPI 同向联动）：

```python
if round_number == 1:
    base_aggressiveness = config.presets[user_prefs.compression_target].aggressiveness
else:
    gap_ratio = (previous_result.total_raster_bytes - raster_budget) / raster_budget

    if gap_ratio > 0.3:      # 超标严重
        base_aggressiveness = min(1.0, prev_aggressiveness + config.aggressiveness_step_large)  # +0.15
    elif gap_ratio > 0:      # 轻微超标
        base_aggressiveness = min(1.0, prev_aggressiveness + config.aggressiveness_step_small)  # +0.08
    elif gap_ratio < -0.15:  # 浪费过多
        base_aggressiveness = max(0.0, prev_aggressiveness - config.aggressiveness_step_small)  # -0.08
    else:
        base_aggressiveness = prev_aggressiveness   # 已达标，通常不会走到这里
```

> **与母本逐图调整的关系**：母本 §8.2 以逐图 `quality_delta（-10/-5/+5）` 和 `dpi_factor（0.8/0.9/1.1）` 表述调整幅度。规范实现是调整全局 base_aggressiveness（上表步长来自 config），经第 4 章公式传导后，单轮参数变化幅度应与母本的 quality ±5~10 / dpi ×0.8~1.1 同量级。调整后仍必须满足只降不升（相对原图）与 quality ∈ [floor, ceiling]。

> **决策理由记录（2026-07-04，用户批准）**：最终采用全局 `base_aggressiveness ± step` 为规范机制、母本逐图数值仅作单轮传导量级校验。理由：**把 N 维搜索空间压缩成 1 维**——若逐图独立调整 quality/DPI，收敛问题是 N 张图 × 2 参数的高维搜索，5 轮上限内无收敛保证；调整单一激进度标量后所有参数同向联动，一维步进搜索在 5 轮内收敛才有保证。

> **prev_aggressiveness 的数据通路（契约修订 2026-07-04）**：decide 把本轮所用值写入 `CompressionPlan.base_aggressiveness`；Phase E 不做任何计算、原样回带到 `CompressionResult.base_aggressiveness`；下一轮 decide 从 `previous_result.base_aggressiveness` 读取。收敛轨迹另记 `PipelineContext.convergence_history`（仅观测，不参与决策）。

### 8.3 收敛期望

- 每一轮的 total_bytes 单调趋近 target，不振荡
- 5 轮内收敛（|gap| < 15%）成功率 > 90%（用 10+ 份不同作品集测试）
- 平均收敛轮数 ≤ 3

### 8.4 收敛失败处理（铁律 7）

5 轮后仍未达标：

- **仍超标** → `PhaseError(code="CONVERGENCE_FAILED")`，向用户返回："目标 X MB 过小，当前最优压缩结果为 Y MB。建议选择更大的目标或标记更多页面为可激进压缩。"
- **浪费过多**（比目标小超过 15%）→ 接受当前结果（宁可小一点也不超标）

任何情况下**不允许静默输出超出目标 15% 以上的文件**。
